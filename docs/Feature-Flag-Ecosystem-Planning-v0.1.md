## 1. Purpose and Context

This document captures the Planning-phase artifacts for a bespoke feature-flag ecosystem that supports producing **multiple adjacent products from a single .NET codebase** across **Android, Windows, iOS, macOS, and Linux**. It is the first iteration. Per the Byrd Process, requirements are expected to be refined as Implementation surfaces defects in this plan ŌĆö that is a feature of the process, not a failure of it.

## 2. V┬▓ Assessment (Viable AND Valuable)

**Valuable?** Yes. The constraint of shipping multiple adjacent products from one codebase creates a permanent need for two distinct kinds of variation: ship-time product identity (which features even exist in this binary) and run-time operational control (gradual rollout, kill switches, regional behavior, A/B experiments). Without a disciplined system, each new product multiplies branching complexity and each incident multiplies time-to-mitigate. A correctly designed flag ecosystem converts both classes of variation into data, where they can be reviewed, audited, and reverted without a redeploy.

**Viable?** Yes, with caveats. The technical risks are bounded and known: AOT-safe expression evaluation on iOS, manifest signing and key rotation, trim-safe SDK packaging for MAUI, and disciplined offline-cache invalidation. None of these are research-grade problems; all have established solutions that we will adopt rather than invent. The product risk ŌĆö that the team builds an admin plane no one uses ŌĆö is mitigated by sequencing the iteration phases to deliver SDK + bundled defaults before any dashboard work.

**V┬▓ == true.** Proceed to detailed planning.

## 3. Glossary and Dimensional Model

These terms are precise; deviations from them in code, schema, or UI are defects.

A **Product** is a shipped, named, store-listed (or otherwise distributable) artifact derived from the single codebase. Product identity is **immutable for a given build** and is stamped at compile time. Examples: "Acme Pro," "Acme Lite," "Acme Education."

A **Release** is a versioned cut of a Product. It carries a semantic version, an immutable Release ID generated by CI, and a release channel (e.g. canary / beta / stable). A Release is also stamped at compile time and never mutates after publication.

The **Evaluation Context** is the structured input to the rule engine when resolving a flag. Product and Release are **required** fields; all other fields are optional but recognized.

The model treats Product and Release as the two required dimensions, but the rule engine accepts **N additional dimensions** carried as ordinary context fields (OS, AppVersion, Locale, DeviceId, UserId, Environment, custom attributes). Nothing in the schema hardwires the count to two ŌĆö Product and Release are simply *required*, not *exclusive*.

A **Flag** is a named, typed variable whose value is computed from the Evaluation Context by an ordered list of Rules. Each Rule pairs a boolean expression with a typed value; the first matching rule wins; if none match, the flag's Default is returned.

A **Manifest** is an immutable, signed bundle of Flag definitions targeting a specific (Product, Release) tuple at a specific point in time. Clients always evaluate against a Manifest, never against the live admin database.

A **Bundled Default Manifest** is the Manifest embedded in the shipped binary at build time. It guarantees the application can boot fully-configured with zero network availability.

A **Remote Override Manifest** is a newer Manifest fetched from the Distribution service post-launch. It is verified against the bundled public key and cached locally. It supersedes the Bundled Default once verified.

## 4. Functional Requirements

The system must, at minimum, do the following.

**FR-1 ŌĆö Compile-time product identity.** The build system shall stamp each produced binary with an immutable ProductId and ReleaseId such that no runtime configuration can change them. Attempting to override these values at runtime shall fail loudly.

**FR-2 ŌĆö Bundled defaults.** Every build shall include a signed default Manifest for its (ProductId, ReleaseId) tuple, embedded as a resource. The application shall be fully functional with no network connectivity using only this manifest.

**FR-3 ŌĆö Remote override.** The SDK shall fetch newer Manifests from the Distribution service on a configurable cadence, verify their signature against an embedded public key, and persist them to a local cache. If verification fails, the new Manifest shall be discarded and the prior valid Manifest retained.

**FR-4 ŌĆö Deterministic evaluation.** Given identical inputs (Manifest, Evaluation Context), the evaluator shall produce identical outputs across all five target platforms. Determinism is testable and tested.

**FR-5 ŌĆö Full ruleset evaluation.** The rule engine shall support a CEL (Common Expression Language) dialect over the Evaluation Context, including equality, comparison, set membership, semantic-version comparison via a custom function, and percentage bucketing via a custom function. CEL is chosen over JSONLogic for its formal grammar, deterministic evaluation semantics, sandbox safety (no implicit type coercion ambiguities), sub-millisecond compile-and-cache cost, and proven scale (Kubernetes admission control, Envoy, Cloud Armor, Cerbos). A documented subset of CEL is supported; user-defined functions are forbidden in v1.

**FR-6 ŌĆö N-dimensional resolution with Product + Release mandatory.** The evaluator shall accept arbitrary context fields. Manifests shall be addressable by (ProductId, ReleaseId) and *may* be further filtered by additional dimensions within rules.

**FR-7 ŌĆö Kill switch.** Every flag shall be marked killable: true | false at definition time. Killable flags shall support a forced-refresh path that bypasses normal cache TTLs.

**FR-8 ŌĆö Exposure tracking.** Each evaluation shall (optionally per flag) emit an exposure event recording flag key, resolved value, the index of the rule that matched, a context fingerprint, and a timestamp. Exposure events shall be buffered offline and uploaded best-effort.

**FR-9 ŌĆö Authoring and audit.** All flag creation, rule changes, and manifest publishes shall be auditable: who, what, when, why. The system shall preserve full history; no destructive edits.

**FR-10 ŌĆö Product-scoping.** Every flag shall declare which Products are permitted to evaluate it. Attempts to read a flag from a non-scoped Product shall return Default and emit a warning.

**FR-11 ŌĆö Multi-environment.** The admin plane shall support the built-in development / staging / production environments plus customer-defined custom environments per the SharpNinja deployment model. Promotion between environments shall be an explicit, audited action.

**FR-12 ŌĆö Schema validation in CI.** The CLI shall validate flag definitions, rule syntax, type safety, and Product-scope correctness in CI, failing builds on violations before any binary ships.

## 5. Non-Functional / Technical Requirements

**TR-1 ŌĆö Target frameworks.** SDK targets `net10.0`, `net10.0-android`, `net10.0-ios`, `net10.0-maccatalyst`, `net10.0-windows10.0.19041.0`, and plain `net10.0` for Linux. Long-term support framework only; no .NET Framework support.

**TR-2 ŌĆö AOT and trim safety.** The SDK and rule evaluator shall be AOT-compatible (iOS, NativeAOT) and trim-safe. This forbids `Reflection.Emit`, dynamic expression compilation, and code generation at runtime. The evaluator shall be a tree-walking interpreter over parsed AST nodes.

**TR-3 ŌĆö Determinism across platforms.** Percentage bucketing shall use a fixed, platform-independent hash (recommend SipHash-2-4 of the concatenation of ProductId || ReleaseId || FlagKey || the bucketing-context value). Floating-point comparisons in rules shall use a documented epsilon or be forbidden entirely in v1.

**TR-4 ŌĆö Signing.** Manifests shall be signed with Ed25519. The public key shall be embedded in the SDK at build time. Key rotation requires a new SDK build; this is acceptable because builds are routine.

**TR-5 ŌĆö Offline-first.** All public SDK calls shall be synchronous and constant-time relative to network state. No public API may block on network I/O. Fetching is a background task; evaluation reads from in-memory state.

**TR-6 ŌĆö Cache hierarchy.** Resolution order on each `Evaluate`: (1) in-memory cached Manifest; (2) on-disk cached Manifest loaded once at startup; (3) bundled-default Manifest. Network fetching only refreshes (1) and (2); evaluation never waits on it.

**TR-7 ŌĆö Telemetry budget.** Exposure events shall be coalesced; the SDK shall not emit more than one network request per N seconds (configurable, default 30) for telemetry under steady-state usage.

**TR-8 ŌĆö Backward compatibility.** The Manifest schema shall be versioned. Newer SDKs shall read older Manifests. Older SDKs shall reject Manifests with a version field higher than they understand and fall back to bundled defaults.

**TR-9 ŌĆö Security.** The Distribution API shall use product-scoped API keys plus device-attestation where the platform supports it (Play Integrity, App Attest). The admin plane shall enforce SSO and per-Product RBAC.

**TR-10 ŌĆö Observability.** The admin plane and Distribution service shall expose Prometheus metrics. The SDK shall expose a diagnostic snapshot API for in-app debug menus.

**TR-11 ŌĆö Dependency Injection and ILogger are non-negotiable at all layers.** Every type in every component shall obtain its collaborators by constructor injection from a `Microsoft.Extensions.DependencyInjection` container, and shall accept `ILogger<TSelf>` (or an equivalent typed logger) wherever logging is performed. Static loggers (`LogManager.GetLogger()`, `Log.ForContext<T>()`, ambient `Trace`/`Debug` calls), service locators, and `new`-ing of injectable dependencies are forbidden. Every public package shall ship `IServiceCollection` extension methods (and, where the target host is MAUI, `MauiAppBuilder` extensions) such that registration is a single call from the consumer's composition root. No public type shall be constructable outside the container; constructors of injectable types are not part of the supported API. Tests resolve subjects from a configured test container, never via direct `new`. The full implications across each layer are described in ┬¦7.

## 6. Component Architecture

The system decomposes into ten components. Each is owned by a clear interface; the interfaces are documented here so implementation can begin via TDD without further design work.

### 6.1 Client SDK Surface

The in-app API surface uses the **OpenFeature .NET public types** ŌĆö `IFeatureClient`, `IFeatureProvider`, `EvaluationContext`, `FlagEvaluationDetails<T>`, `Reason`, `ErrorCode` ŌĆö but **does not use OpenFeature's static `Api.Instance` global**. The OpenFeature NuGet ships its types and its default `FeatureClient` implementation together; we adopt the former and supply our own implementation of the latter. Consumers `[Inject] IFeatureClient` and `[Inject] IFeatureProvider` directly from the container; the `Api.Instance` singleton is never touched in product code, in tests, or in our own internals.

The benefit of keeping OpenFeature's public types: consumer code is written against a well-documented, vendor-neutral API; OpenFeature `Hook<T>` instances can be registered as ordinary DI singletons and applied by our client implementation; and if the project ever wants to migrate to a different OpenFeature provider implementation, application code does not change. The cost is exactly one NuGet dependency on `OpenFeature` for the public types.

Three NuGet packages are produced:

`SharpNinja.FeatureFlags.Abstractions` ŌĆö the options record (`SharpNinjaFeatureFlagOptions`), the bespoke admin interface (`ISharpNinjaFeatureFlagAdmin`), the diagnostic types, and the `ManifestSource` factory. Depends on `OpenFeature` (for `EvaluationContext`/etc.) and `Microsoft.Extensions.Logging.Abstractions` ŌĆö nothing else. Consumed by application code at compile time without dragging in implementation.

`SharpNinja.FeatureFlags` ŌĆö the implementation: `SharpNinjaFeatureFlagProvider : IFeatureProvider`, our DI-resident `SharpNinjaFeatureClient : IFeatureClient`, the manifest store, the CEL evaluator, the signature verifier, the distribution fetcher, the hosted-service initializer, and the `AddSharpNinjaFeatureFlags(ŌĆ”)` extension. Depends on `SharpNinja.FeatureFlags.Abstractions` and the standard `Microsoft.Extensions.*` packages.

`SharpNinja.FeatureFlags.Build` ŌĆö MSBuild integration and source generator. See ┬¦6.8.

Registration and consumption are DI-driven end-to-end (per TR-11). The provider, client, manifest store, rule evaluator, fetcher, and signature verifier are all container-resolved; nothing is `new`-ed by consumer code.

**ASP.NET Core / generic host / console:**


`AddSharpNinjaFeatureFlags` is exposed by `SharpNinja.FeatureFlags` and internally registers, **as singletons in the consumer's container**:

- `IFeatureProvider` ŌåÆ `SharpNinjaFeatureFlagProvider` (our implementation of OpenFeature's provider interface, holding the manifest store, CEL evaluator, signature verifier, and fetcher).
- `IFeatureClient` ŌåÆ `SharpNinjaFeatureClient` (our implementation, takes `IFeatureProvider` and the registered `IEnumerable<Hook>` collection through its constructor; never reads `Api.Instance`).
- `IManifestStore`, `IRuleEvaluator`, `ISignatureVerifier`, `IDistributionFetcher` (the last via `IHttpClientFactory`), `IExposureBuffer`, and `ISharpNinjaFeatureFlagAdmin`.
- An `IHostedService` (`FeatureFlagInitializerHostedService`) that performs the synchronous bundled-default load on startup, kicks off the first background remote-refresh, and signals readiness before the host completes `StartAsync`.

The extension does **not** call `OpenFeature.Api.Instance.SetProvider(...)`. The global is never written and never read by any code we ship. Hooks are registered as ordinary DI services (`services.AddSingleton<Hook, MyHook>()`); our `SharpNinjaFeatureClient` resolves them by injecting `IEnumerable<Hook>` and applies them in the same lifecycle order OpenFeature's reference client does.

The `SharpNinjaFeatureFlagProvider` automatically injects `ProductId` and `ReleaseId` into every Evaluation Context from the compile-time build stamp; attempts to set them via the consumer's `EvaluationContext` are ignored with a diagnostic warning. This preserves FR-1 (compile-time product identity) while keeping the consumer-facing API identical to OpenFeature's.

The bespoke admin surfaces ŌĆö `RefreshAsync`, `ForceRefreshAsync`, `GetDiagnostics`, `ManifestUpdated` event ŌĆö are exposed through the injectable `ISharpNinjaFeatureFlagAdmin` service, so debug menus and ops tooling participate in the same DI graph as the rest of the app. Application code that uses these features takes a hard dependency on `SharpNinja.FeatureFlags.Abstractions`; application code that only evaluates flags depends solely on `OpenFeature` types and remains theoretically portable.

#### 6.1.1 Why this is sane

OpenFeature's `Api.Instance` is a process-wide singleton that holds the active provider, the hook list, the evaluation-context evaluator, and the global event bus. In a strict-DI codebase this is a problem on four axes: test isolation (state leaks between integration tests in the same process), parallel test execution (writes to the global race), lifetime control (the global outlives any `IServiceProvider`), and discoverability (a dependency that doesn't appear in a constructor is invisible to architectural analysis). By implementing `IFeatureClient` ourselves and registering it in DI, all four problems disappear and TR-11 is satisfied without exception. The OpenFeature API contract is honoured because the public types and method semantics are unchanged; only the wiring is ours.

### 6.2 Rule Engine (`SharpNinja.FeatureFlags.Evaluation`)

A pure library with no I/O. Takes a parsed Manifest and an Evaluation Context, returns a typed evaluation outcome. AOT-safe tree-walking interpreter over a parsed CEL AST. No `Reflection.Emit`, no `System.Linq.Expressions.Compile()`, no Roslyn ŌĆö the AST is interpreted node-by-node, which is mandatory for iOS and NativeAOT and remains fast because rules are tiny and the AST is cached after the first parse.

Exposed surface:

```csharp
public interface IRuleEvaluator
{
    EvaluationOutcome Evaluate(
        FlagDefinition flag,
        OpenFeature.Model.EvaluationContext context);
}
```

The CEL subset supported in v1: scalar literals (bool, int, double, string, null), list and map literals, the standard CEL operators (`==`, `!=`, `<`, `<=`, `>`, `>=`, `&&`, `||`, `!`, `in`, `?:`), arithmetic on numerics, string concatenation, indexing (`map[key]`, `list[i]`), field access (`context.ProductId`), and a fixed table of custom functions:

- `semver_satisfies(version, constraint)` ŌĆö semver-range match.
- `bucket(seed, key, denominator)` ŌĆö deterministic percentage bucketing using SipHash-2-4. Returns an integer in `[0, denominator)`.
- `version_compare(a, b)` ŌĆö three-way semver compare.

User-defined functions are forbidden in v1. Regular expressions are forbidden in v1 (regex on untrusted input is a denial-of-service vector). CEL macros (`exists`, `all`, `filter`, `map`) are permitted but constrained to a maximum iteration count per evaluation (default 1000) to bound worst-case cost.

Parsed CEL programs are cached per-FlagDefinition at Manifest load time, so steady-state evaluation is pure interpretation against a pre-built AST ŌĆö typically sub-microsecond.

### 6.4 Distribution Service

Stateless HTTP service fronted by a CDN. Three endpoints in v1:

- `GET /v1/manifest/{productId}/{releaseId}` ŌĆö returns latest signed Manifest for the (Product, Release).
- `GET /v1/manifest/{productId}/{releaseId}/delta?since={etag}` ŌĆö returns delta or 304.
- `POST /v1/exposure` ŌĆö accepts batched exposure events.

Authentication is per-Product API keys carried in `Authorization: Bearer ŌĆ”`. Rate limiting is per-key. Caching headers (`ETag`, `Cache-Control`) are correct because the CDN does the actual work.

### 6.5 Admin Plane (`SharpNinja.FeatureFlags.Admin`)

A Docker-hosted ASP.NET Core **Blazor Web App** (.NET 10, Interactive Server render mode) hosts the entire administration surface, including flag CRUD and rule authoring. Persistence is provider-agnostic Entity Framework Core; the data layer is described in ┬¦6.6. The same host process exposes a small JSON API (`/api/v1/ŌĆ”`) for programmatic access by `flagctl` and CI integrations; UI and API share the application services layer so business rules are defined once.

Render-mode rationale: Interactive Server gives real-time validation, sub-second rule-preview round-trips against synthetic Evaluation Contexts, live audit-log streams, and live manifest-diff displays without a separate API/state-management layer. The admin plane is an internal tool with a known concurrent-user ceiling, so SignalR connection cost is acceptable. Public-facing or high-fan-out pages (none planned for v1) would warrant Interactive Auto or pure Server-Side Rendering instead.

Component breakdown:

- **Pages** ŌĆö `Flags/Index` (Product ├Ś Release pivot grid), `Flags/Edit` (definition + rule editor with CEL syntax highlighting and inline validation), `Manifests/Build` (compose-and-sign workflow), `Manifests/Promote` (dev ŌåÆ staging ŌåÆ prod), `Audit/Index` (immutable history), `Products/Index`, `Releases/Index`, `Settings/Keys` (signing key rotation).
- **Rule editor** ŌĆö Monaco editor (loaded via JS interop) configured with a CEL language definition. Each keystroke parses against the same CEL grammar the SDK uses; parse errors are highlighted inline. A side panel runs the rule against a configurable set of synthetic Evaluation Contexts and shows the resolved value live.
- **Preview-before-publish** ŌĆö every Manifest publish requires a diff review screen that shows every flag whose resolved value changes for any of a saved set of "canary contexts" the team maintains. Publishing without acknowledging the diff is not possible.
- **Authentication** ŌĆö `Microsoft.AspNetCore.Authentication.OpenIdConnect` against the customer's IdP. No local accounts.
- **Authorization** ŌĆö policy-based, scoped by (Product, Environment) tuple. Roles: `Viewer`, `Editor`, `Publisher`, `KeyAdmin`. Publishing to production requires an `Editor` *plus* a `Publisher` approval (two-person rule).
- **Audit** ŌĆö every write goes through a single `IAuditWriter` that records actor, action, before/after, and reason text. Audit rows are append-only at the database level (revoke `UPDATE`/`DELETE` on the audit table from the application role).

The admin plane never serves Manifests to client SDKs directly; that is the Distribution service's job (┬¦6.4). The admin plane *produces* signed Manifests and *pushes* them to the Distribution service's origin storage.

Shared library: a `SharpNinja.FeatureFlags.Schema` package contains the Flag/Rule/Manifest/EvaluationContext types, the CEL parser, and the manifest signer. It is referenced by the SDK provider, the admin plane, the Distribution service, and `flagctl` ŌĆö so a schema change ripples through every component at compile time.

### 6.6 Data Layer (`SharpNinja.FeatureFlags.Admin.Data` and per-provider migration assemblies)

Persistence uses **Entity Framework Core** with a provider-agnostic `DbContext` plus one **migration assembly per supported database provider**. v1 supported providers are **PostgreSQL** and **SQL Server**. Other database providers may be reconsidered after v1, but the v1 Docker composition and validation gates target PostgreSQL plus SQL Server.

#### 6.6.1 Schema shape

The schema is a conventional, normalized, multi-table relational design in **Fourth Normal Form (4NF)**. Every relation satisfies BCNF, and every non-trivial multi-valued dependency has been decomposed into its own junction table. Foreign-key integrity is enforced at the database layer wherever the provider supports it.

**Primary entities (one row per real-world entity):**

- `Products` ŌĆö immutable `ProductId`, display name, lifecycle state.
- `Releases` ŌĆö `(ProductId, ReleaseId)` keyed; FK to `Products`; semantic version, channel (`canary`/`beta`/`stable`), publication timestamp.
- `Environments` ŌĆö built-in `Development`, `Staging`, `Production`, plus custom-defined regional or customer environments.
- `Flags` ŌĆö flag identity; key, type, default, lifecycle state (`Proposed`/`Active`/`Sunsetting`/`Retired`), expiration date, `Killable` boolean. **No multi-valued columns.**
- `FlagRules` ŌĆö ordered list of rules per flag; FK to `Flags`; CEL source text, parsed-AST cache key, typed value, rule ordinal. **No multi-valued columns** ŌĆö environment scope and channel scope are externalized (see junctions below).
- `Manifests` ŌĆö one row per `(ProductId, ReleaseId, EnvironmentId, Version)`; FK to `Products`, `Releases`, `Environments`; signed-blob reference, signature, signing-key ID.
- `ManifestEntries` ŌĆö flag-state snapshot inside each manifest version; FK to `Manifests` and `Flags`.
- `SigningKeys` ŌĆö Ed25519 keypair metadata; private-key material lives in the host's secret store, never in this table.
- `ApiKeys` ŌĆö per-Product API key identity; FK to `Products`; hashed value, rotation timestamps. **No multi-valued columns.**
- `AuditEvents` ŌĆö append-only history; actor, action, target entity type and ID, before/after JSON payloads, reason text, timestamp.
- `ExposureEvents` ŌĆö retained exposure records keyed by Product, Release, Environment, optional Tenant, flag key, context fingerprint, and timestamp.
- `ExposureRetentionPolicies` ŌĆö user-defined retention periods scoped by Product, Environment, and optional Tenant.
- `Tenants` ŌĆö tenant identity and lifecycle state for multi-tenant deployments.
- `Tags` ŌĆö tag identity.
- `Users`, `Roles` (if not fully delegated to the IdP).

**Junction tables (one per independent multi-valued dependency, per 4NF):**

- `FlagOwners(FlagId, OwnerId)` ŌĆö a flag may have multiple owners.
- `FlagTags(FlagId, TagId)` ŌĆö a flag may carry multiple tags.
- `FlagLinkedTickets(FlagId, TicketSystem, TicketKey)` ŌĆö a flag may link to multiple external tickets across multiple systems.
- `FlagProductScopes(FlagId, ProductId)` ŌĆö enforces FR-10; a flag may be scoped to multiple products.
- `FlagRuleEnvironments(FlagRuleId, EnvironmentId)` ŌĆö a rule may apply in multiple environments.
- `FlagRuleReleaseChannels(FlagRuleId, ReleaseChannel)` ŌĆö a rule may apply across multiple release channels. **This is deliberately a separate table from `FlagRuleEnvironments`** ŌĆö environment-scope and channel-scope are independent MVDs of `FlagRuleId`, and 4NF forbids combining independent MVDs in a single relation.
- `ManifestApprovals(ManifestId, ApproverId, ApprovedAt)` ŌĆö supports the two-person production-publish rule; each approval is its own row.
- `ApiKeyScopes(ApiKeyId, ScopeId)` ŌĆö an API key may carry multiple endpoint/permission scopes.
- `ApiKeyAllowedCidrs(ApiKeyId, Cidr)` ŌĆö an API key may permit multiple source CIDR ranges. Independent of `ApiKeyScopes` and therefore its own table.
- `UserRoleAssignments(UserId, RoleId, Scope)` ŌĆö a user may hold multiple roles, scoped per (Product, Environment).

The decomposition is exhaustive for v1: any column whose value is a set, list, or multi-cardinality reference is hoisted into a dedicated junction. Future entities must pass the same audit before merging.

Hot-path indexes: composite `(ProductId, ReleaseId)` on `Manifests`; `(FlagId, Ordinal)` on `FlagRules`; `(Timestamp DESC)` plus `(ActorId, Timestamp DESC)` on `AuditEvents`; covering indexes on the high-cardinality side of every junction (`FlagOwners(OwnerId)`, `FlagTags(TagId)`, etc.) to support reverse lookups. Provider-specific index syntax (filtered indexes, `INCLUDE` columns, expression indexes) is hidden behind the dialect helper described below.

#### 6.6.2 Migration discipline ŌĆö one table per migration class

**Every migration class shall mutate exactly one table.** A migration that adds a column to `Flags` does not also seed `Releases`. A migration that creates a new table does not also alter an existing one. Multi-table changes are decomposed into multiple, ordered migration classes. This is the project's load-bearing migration discipline and the heart of the per-provider migration story.

Rationale: atomic review and rollback (a defective migration affects one logical surface, not many); merge-conflict isolation (table-X migrations don't collide with table-Y migrations); legible per-table change history under `git log`; deliberate ordering forced for dependent schema changes (you cannot bundle the dependency and lose the chain).

Naming convention: `{YYYYMMDD}_{NNN}_{Verb}_{TableName}[_{Variant}].cs`, e.g. `20260601_001_Create_Products.cs`, `20260601_002_Create_Releases.cs`, `20260615_001_AddColumn_Flags_ExpirationDate.cs`. The table name is encoded in the class name and matched against an internal registry.

Enforcement: a unit test in `SharpNinja.FeatureFlags.Admin.Data.Tests` reflects over every type that derives from `Microsoft.EntityFrameworkCore.Migrations.Migration` in every provider assembly, executes `Up()` and `Down()` against an `IModel`-aware fake `MigrationBuilder`, and asserts that the set of distinct table identifiers touched by the captured operations has cardinality exactly one. The test runs in CI; violations fail the build. A small attribute (`[MigrationTable("TableName")]`) lets authors declare the intended table; raw `Sql()` operations require the attribute and are verified by string-matching the SQL against the declared table name.

```csharp
[MigrationTable("Flags")]
public partial class AddColumn_Flags_ExpirationDate : Migration
{
    protected override void Up(MigrationBuilder mb)
    {
        mb.AddColumn<DateTime?>(
            name: "ExpirationDate",
            table: "Flags",
            type: ProviderDialect.Current.TimestampUtc,
            nullable: true);
    }
    protected override void Down(MigrationBuilder mb) =>
        mb.DropColumn("ExpirationDate", "Flags");
}
```

The single-table discipline does **not** forbid cross-table dependencies in production deployments ŌĆö it forbids combining them in a single migration class. A feature that adds a column to `Flags` and a backfill index on `FlagRules` is shipped as two migrations applied in declared order.

Each v1 provider package depends on the shared data package, references the appropriate EF Core provider (`Npgsql.EntityFrameworkCore.PostgreSQL` or `Microsoft.EntityFrameworkCore.SqlServer`), and contains its own `Migrations/` folder. The admin-plane composition root selects the right assembly at runtime based on configuration:

```csharp
services.AddDbContext<AdminDbContext>(options =>
{
    var providerName = config["Database:Provider"];      // "Postgres" | "SqlServer"
    var connectionString = config.GetConnectionString("Admin");

    switch (providerName)
    {
        case "Postgres":
            options.UseNpgsql(connectionString, npgsql =>
                npgsql.MigrationsAssembly("SharpNinja.FeatureFlags.Admin.Data.Postgres")
                      .MigrationsHistoryTable("__SharpNinjaMigrations"));
            break;

        case "SqlServer":
            options.UseSqlServer(connectionString, sql =>
                sql.MigrationsAssembly("SharpNinja.FeatureFlags.Admin.Data.SqlServer")
                   .MigrationsHistoryTable("__SharpNinjaMigrations"));
            break;
        default:
            throw new ConfigurationException($"Unsupported provider: {providerName}");
    }
});
```

The migrations-history table is renamed `__SharpNinjaMigrations` on every provider so the history convention is identical ŌĆö no `[dbo].[__EFMigrationsHistory]` versus `public."__EFMigrationsHistory"` ambiguity in operational tooling.

Migrations are conceptually parallel across providers: the same logical change appears as a `{date}_{seq}_{verb}_{table}.cs` class in each provider's assembly. A shared `IProviderDialect` exposes operation-builder helpers for the patterns that vary by provider ŌĆö JSON column type, timestamp type with timezone semantics, row-version concurrency, computed columns, append-only constraint emission ŌĆö so the per-provider divergence is contained to a small surface and equivalent migrations remain visually parallel across assemblies.

#### 6.6.4 JSON column portability

A small number of columns store JSON payloads ŌĆö `AuditEvents.Before`, `AuditEvents.After`, `FlagRules.CompiledAstCache`, `ManifestEntries.ResolvedValue`. These are the only places where provider behavior genuinely diverges. The data layer addresses this with two rules:

First, **application code never writes JSON path filters**; repository methods encapsulate provider-specific JSON access. `Where(e => EF.Functions.JsonValue(...) == "x")` is a code-review violation enforced by an analyzer.

Second, **anything that needs to be filterable becomes a typed column**, not a JSON probe. JSON storage is for round-tripping payloads that are read whole and rendered, never for filtering.

#### 6.6.5 Append-only audit enforcement

The `AuditEvents` table is enforced append-only at the database role level. The application's runtime role is granted `INSERT` and `SELECT` only; `UPDATE` and `DELETE` are revoked. The role grants are emitted by a dedicated, single-table migration class that targets `AuditEvents` exclusively, consistent with the discipline in ┬¦6.6.2. On v1 providers this is implemented with provider-native primitives ŌĆö `REVOKE` on the PostgreSQL role and `DENY UPDATE, DELETE` on the SQL Server role ŌĆö while preserving an identical application contract.

### 6.9 Source-Generated Flag Binding (`SharpNinja.FeatureFlags.Build`, additional generators)

Ships in the same `SharpNinja.FeatureFlags.Build` NuGet as ┬¦6.8 but is a distinct capability: a set of Roslyn incremental source generators that bind feature flags to types and members declaratively via attributes. The goal is to move flag adoption from imperative `if (flags.GetBoolean(...))` scatter to a compile-time-validated contract ŌĆö so flag use is discoverable, statically checkable, and harder to forget about (the latter is the explicit antidote the project name calls for).

#### 6.9.1 Attribute surface

The attributes live in `SharpNinja.FeatureFlags.Abstractions` so consumers reference them without a runtime dependency on the implementation package.

```csharp
namespace SharpNinja.FeatureFlags;
```

#### 6.9.2 Generator behaviors

**Class-level boolean gating: `[FeatureFlag(key)]` + `[FeatureFlagFallback(typeof(I))]`.**
A class attributed with `[FeatureFlag("checkout.v2.enabled")]` that implements an interface `ICheckoutService`, paired with another class attributed `[FeatureFlagFallback(typeof(ICheckoutService))]`, causes the generator to emit a partial `IServiceCollection` extension:

```csharp
// emitted by generator
public static partial class GeneratedFeatureFlagRegistrations
{
    public static IServiceCollection AddCheckoutServiceFeatureGated(this IServiceCollection services)
    {
        services.AddTransient<ICheckoutService>(sp =>
        {
            var client = sp.GetRequiredService<ISharpNinjaFeatureClient>();
            var enabled = client.GetBoolean("checkout.v2.enabled", defaultValue: false).Value;
            return enabled
                ? new CheckoutServiceV2(/* generator-resolved ctor args via DI */)
                : new CheckoutServiceV1(/* generator-resolved ctor args via DI */);
        });
        return services;
    }
}
```

Constructor arguments are resolved by emitting explicit `sp.GetRequiredService<T>()` calls for each parameter ŌĆö no reflection, no `ActivatorUtilities`, AOT-safe.

**Class-level variant selection: `[FeatureFlagVariant(key, value)]`.**
Two or more classes implementing the same interface, each attributed with a different `VariantValue` for the same `FlagKey`, generate a string-valued variant resolver. The flag must be of type `string` (validated against the manifest); attribute placement on classes with non-string flags emits `SNFF0002`. Duplicate variant values emit `SNFF0009`.

**Property / field value binding: `[FeatureFlagValue(key)]`.**
Applied to a `partial` property of a class that injects `ISharpNinjaFeatureClient`, the generator emits the property's getter as a call to `_flagClient.Get<T>(key, default)`. The property's CLR type must match the flag's declared type. If the property is not `partial`, the generator emits `SNFF0006`; if the containing class has no `ISharpNinjaFeatureClient` constructor parameter, `SNFF0007`.

**Method gating: `[FeatureFlagGate(key)]`.**
Applied to a `partial` method of a class that injects `ISharpNinjaFeatureClient`. The developer declares a partial method header with no body and an implementation method named `{MethodName}_Impl` containing the real logic; the generator emits the gating wrapper:

```csharp
// developer writes:
public partial class ApiController
{
    [FeatureFlagGate("api.beta.endpoint")]
    public partial Task<IActionResult> NewEndpoint(CancellationToken ct);

    private async Task<IActionResult> NewEndpoint_Impl(CancellationToken ct) { /* real work */ }
}

// generator emits:
public partial class ApiController
{
    public async partial Task<IActionResult> NewEndpoint(CancellationToken ct)
    {
        var result = _flagClient.GetBoolean("api.beta.endpoint", false);
        LogGateEvaluated(_logger, "api.beta.endpoint", result.Value, result.Reason);
        if (!result.Value) return HandleDisabled(); // per DisabledBehavior on the attribute
        return await NewEndpoint_Impl(ct);
    }

    [LoggerMessage(EventId = 8101, Level = LogLevel.Debug,
        Message = "Feature gate {FlagKey} resolved to {Enabled} (reason {Reason})")]
    static partial void LogGateEvaluated(ILogger logger, string flagKey, bool enabled, EvaluationReason reason);
}
```

The `_flagClient` and `_logger` field references resolve to constructor-injected fields of the containing class. The generator detects whether the class already injects `ISharpNinjaFeatureClient` and `ILogger<TSelf>`; if either is missing, it either (a) augments the existing constructor by emitting an additional `partial` constructor part, or (b) emits a diagnostic (`SNFF0007`) demanding the consumer inject them. The first behavior is the default to minimize friction; the second is opt-in via an MSBuild property for stricter codebases.

C# 12+ **interceptors** are a future-but-not-v1 enhancement that would let the generator gate ordinary (non-`partial`) method *call sites* without source modification. The doc tracks interceptors as a v2 capability once they leave preview; v1 uses the `partial`-method pattern exclusively because it is stable, public, and AOT-safe today.

#### 6.9.3 Compile-time validation

The generator reads the same flag-manifest JSON that ┬¦6.8 embeds as a bundled-default resource. The file is exposed to the generator as an `AdditionalFiles` MSBuild item:

```xml
<ItemGroup>
  <AdditionalFiles Include="flags/flags.json" />
</ItemGroup>
```

(The MSBuild target in `SharpNinja.FeatureFlags.Build` adds this automatically when the canonical path is used.) The generator parses the manifest once per compilation via `IncrementalValueProvider` so subsequent incremental builds are fast.

Diagnostics emitted, with stable IDs in the `SNFF0xxx` band:

- **`SNFF0001` Error** ŌĆö `[FeatureFlag("x")]` references a flag key `x` not present in the project's flag manifest.
- **`SNFF0002` Error** ŌĆö Flag type mismatch: e.g., `[FeatureFlag]` (boolean-gating) on a class whose referenced flag is of non-boolean type.
- **`SNFF0003` Error** ŌĆö Current `ProductId` (from the build stamp) is not in the referenced flag's `ProductScope`.
- **`SNFF0004` Error** ŌĆö `[FeatureFlag]`-attributed class does not implement an interface that another `[FeatureFlagFallback]`-attributed class targets.
- **`SNFF0005` Error** ŌĆö `[FeatureFlagFallback(typeof(I))]` references an interface `I` for which no `[FeatureFlag]`-attributed class exists in this compilation.
- **`SNFF0006` Error** ŌĆö `[FeatureFlagValue]` or `[FeatureFlagGate]` applied to a non-`partial` member.
- **`SNFF0007` Error** ŌĆö Containing class has no `ISharpNinjaFeatureClient` constructor parameter and strict-mode validation is enabled.
- **`SNFF0008` Warning** ŌĆö Referenced flag is past its `ExpirationDate` (configurable to error via `<TreatFeatureFlagExpirationAsError>true</TreatFeatureFlagExpirationAsError>`).
- **`SNFF0009` Error** ŌĆö Multiple variants share the same `VariantValue` for the same `FlagKey`.
- **`SNFF0010` Warning** ŌĆö Flag is in `Sunsetting` lifecycle state; usage should be migrated.
- **`SNFF0011` Info** ŌĆö Flag has not been evaluated in any telemetry sample in the last 90 days (requires admin-plane round-trip; emitted only when an MSBuild property enables it).

Each diagnostic links to documentation; the generator emits structured help URLs (`https://docs.sharpninja.example/feature-flags/diagnostics/SNFF0001`) so IDE quick-fix info populates correctly.

The expiration and sunsetting diagnostics (`SNFF0008`, `SNFF0010`, `SNFF0011`) are the load-bearing feature-creep-management affordances: they turn stale flags from invisible cruft into visible build noise that costs to ignore.

#### 6.9.4 DI and ILogger compliance

All emitted code complies with TR-11 and ┬¦7. The generator never emits a static cache, a singleton field, or a `new` of an injectable type. The emitted DI extensions always resolve through the consumer's container. Every emitted code path that logs uses `[LoggerMessage]` source-generated methods with event IDs in the 8000ŌĆō8999 band (consistent with ┬¦7.2). Generated code is fully trim-safe and AOT-safe ŌĆö no reflection, no `Expression.Compile`, no `Activator.CreateInstance(Type)`.

#### 6.9.5 What the generator deliberately does *not* do

It does not produce runtime IL weaving, dynamic proxies, or `Castle.DynamicProxy`-style interception. All gating is compile-time-resolved or call-site-resolved; no runtime monkey-patching.

It does not generate code that auto-registers types in DI without an explicit consumer opt-in. Every emitted `IServiceCollection` extension must be called by the consumer's composition root. This preserves the property from ┬¦7 that no type is constructable outside an explicit container registration the consumer can see.

It does not gate across assembly boundaries via attributes on types it does not own. Attributes apply only to types declared in the consuming compilation.

### 6.10 CQRS Dispatcher Integration (`SharpNinja.FeatureFlags.Cqrs`, adapted from `f:\github\McpServer\`)

CQRS and feature flags are mutually reinforcing patterns. CQRS routes every business operation through a single dispatch chokepoint; feature flags need a uniform enforcement point to avoid `if (flags.GetBoolean(...))` blooming across the codebase. Combining them turns flag enforcement into a dispatcher concern that fires once per operation with the full command context available ŌĆö the cleanest possible blast radius for any flag check.

The CQRS layer for this ecosystem is **not built from scratch**. An existing, in-house CQRS system already lives in `f:\github\McpServer\`, authored as part of the Byrd Process MCP Server work. v1 of `SharpNinja.FeatureFlags.Cqrs` is a **vendored copy of that codebase, renamed and adapted** rather than a greenfield build. This preserves architectural continuity with the broader Byrd ecosystem, eliminates a re-implementation tax, and reduces v1 scope materially.

#### 6.10.1 Import strategy: vendor-folder copy with namespace rename

The chosen import model for v1 is a **vendor-folder copy**:

1. The relevant projects under `f:\github\McpServer\` (the CQRS dispatcher, handler/behavior contracts, pipeline composition, registration helpers, and their existing test suite) are copied into this repository under `src/SharpNinja.FeatureFlags.Cqrs/` and `tests/SharpNinja.FeatureFlags.Cqrs.Tests/`.
2. The root namespace is renamed mechanically (Roslyn-driven find/replace plus targeted manual review for string literals in tests and diagnostics) to `SharpNinja.FeatureFlags.Cqrs.*`.
3. The upstream link is **severed at v1**. Future divergence is owned by this repository. A revisit of the upstream relationship ŌĆö selective backports to McpServer, periodic re-sync, or a managed hard-fork with sync tooling ŌĆö is an open question for v0.2 and beyond, deferred until the v1 adaptation has stabilized.

Alternatives considered: git submodule (rejected for v1 because every flag-aware adaptation becomes a multi-repo concern that blocks shipping); hard fork with periodic upstream sync (rejected for v1 on the same grounds, viable for v2 if the upstream system continues evolving in ways this project wants to track); leaving the code in McpServer and referencing it as a project reference across repos (rejected as fragile under independent versioning).

##### 6.10.2.2 Source inventory (recorded 2026-05-12)

Directory access was granted to `F:\GitHub\McpServer\` and the actual source has been read. The CQRS project is `src/McpServer.Cqrs/` and is exactly 14 hand-written `.cs` files targeting `net9.0` and `net10.0`. The full surface area:

**Markers (no methods):**
- `ICommand<TResult>` ŌĆö write-side marker.
- `IQuery<TResult>` ŌĆö read-side marker.

**Handlers (split intentionally):**
- `ICommandHandler<in TCommand, TResult> where TCommand : ICommand<TResult>` ŌĆö single method `Task<Result<TResult>> HandleAsync(TCommand command, CallContext context)`.
- `IQueryHandler<in TQuery, TResult> where TQuery : IQuery<TResult>` ŌĆö single method `Task<Result<TResult>> HandleAsync(TQuery query, CallContext context)`.
- **Note:** the handler signature takes `CallContext` as the second parameter rather than `CancellationToken`. The CT lives on `CallContext.CancellationToken`.

**Dispatcher:**
- `IDispatcher.SendAsync<TResult>(ICommand<TResult> command, CancellationToken ct = default) ŌåÆ Task<Result<TResult>>`.
- `IDispatcher.QueryAsync<TResult>(IQuery<TResult> query, CancellationToken ct = default) ŌåÆ Task<Result<TResult>>`.
- Concrete `Dispatcher` also implements `ILoggerProvider`, exposing `ActiveContexts` and `RecentDispatches` (bounded to 200) for diagnostics.

**Pipeline behaviors:**
- `IPipelineBehavior.HandleAsync<T>(object request, CallContext context, Func<Task<Result<T>>> next) ŌåÆ Task<Result<T>>`.
- Weakly typed on the request (`object`), strongly typed on the result. Behaviors execute in registration order, outermost first. Short-circuit by returning a `Result<T>` without calling `next`.

**Return type:**
- `Result<T>` ŌĆö value-typed monad with `Success(T)`, `Failure(string)`, `Failure(Exception)`, `Failure(string, Exception)`, plus `Bind<TNext>`, `Map<TNext>`, `GetValueOrDefault(T fallback)`. Non-generic `Result` exists for void-like commands.

**Request envelope:**
- `CallContext` ŌĆö sealed class implementing `ILogger` and `IDisposable`. Carries `Correlation` (`CorrelationId`), `Started` (`DateTimeOffset`), `Elapsed`, `OperationName`, `UserId`, `UserName`, `Roles` (`IReadOnlyList<string>`), `Entries` (`IReadOnlyCollection<LogEntry>` of every log captured during the dispatch), `Properties` (`ConcurrentDictionary<string, object?>` with invariant-case keys), and `CancellationToken`. The `ILogger` implementation captures log entries into the per-dispatch `Entries` bag, separately from any host loggers ŌĆö exactly the audit substrate we want for flag-evaluation correlation.

**Opt-in marker:**
- `IHasTimeout` ŌĆö a request can implement this to set its own dispatch timeout, which the dispatcher honours by creating a linked CTS.

**Registration:**
- `services.AddCqrsDispatcher()` ŌĆö singleton.
- `services.AddCqrsLoggerProvider()` ŌĆö registers `Dispatcher` as `ILoggerProvider` (via `DeferredDispatcherLoggerProvider` to avoid the documented circular dependency Dispatcher ŌåÆ ILogger ŌåÆ ILoggerFactory ŌåÆ ILoggerProvider ŌåÆ Dispatcher).
- `services.AddCqrsHandlers(params Assembly[])` ŌĆö reflective assembly scan that finds implementations of `ICommandHandler<,>` and `IQueryHandler<,>` and registers them as transient.
- `services.AddCqrsBehavior<T>()` ŌĆö registers an `IPipelineBehavior` as transient.
- `services.AddCqrs(params Assembly[])` ŌĆö convenience: dispatcher + handler scan.

**TR-11 conformance:** uniform. Every public type takes its collaborators by constructor injection. `ILogger<Dispatcher>` is injected. There are no static loggers, no service locators, no global state. The deferred logger-provider pattern explicitly mitigates a documented DI ordering hazard. Zero remediation work required before flag awareness is added.

**Apache-2.0 obligations identified:** preserve the repository's `LICENSE` and any `NOTICE` file, retain per-file copyright headers (the inspected files do not currently carry inline copyright headers ŌĆö confirm whether the original LICENSE notice suffices per ┬¦4 of Apache 2.0, or whether headers need to be added during vendoring), and produce a `THIRD-PARTY-NOTICES.md` in the feature-flag repository attributing the upstream project.

##### 6.10.2.3 Adaptation plan (we own the code; we modify what needs modifying)

Vendoring this project means the code becomes ours. The adaptations below are deliberate upstream changes, not workarounds. Each is minimal, additive in the sense that nothing existing breaks, and surfaces the new behavior through a clean extension point rather than through a hack.

**Adaptation 1: `CallContext` unsealed; `IDispatchContextFactory` introduced; `SharpNinjaFlagCallContext` provided.**

`CallContext` ships from upstream as `sealed class`. The vendored copy removes `sealed` (one keyword) and is otherwise byte-identical. The dispatcher gains one new constructor parameter ŌĆö `IDispatchContextFactory contextFactory` ŌĆö and replaces its `new CallContext { ŌĆ” }` line with `contextFactory.Create(operationName: requestType.Name, cancellationToken: ct)`. The factory interface is one method:

```csharp
public interface IDispatchContextFactory
{
    CallContext Create(string operationName, CancellationToken cancellationToken);
}
```

A default registration `services.AddSingleton<IDispatchContextFactory, DefaultDispatchContextFactory>()` lives in the core CQRS package and produces plain `CallContext` instances. The flag-aware package's DI extension calls `services.Replace(ServiceDescriptor.Singleton<IDispatchContextFactory, SharpNinjaFlagDispatchContextFactory>())`, and that factory returns `SharpNinjaFlagCallContext` instances:

```csharp
public sealed class SharpNinjaFlagCallContext : CallContext
{
    public SharpNinjaFlagCallContext(string productId, string releaseId,
                               DeploymentEnvironment environment,
                               string operationName, CancellationToken ct)
        : base()
    {
        OperationName     = operationName;
        CancellationToken = ct;
        ProductId         = productId;
        ReleaseId         = releaseId;
        Environment       = environment;
        IsCommand         = operationName.StartsWith("ICommand") || operationName.EndsWith("Command");
        IsQuery           = !IsCommand;
    }

    public string                ProductId   { get; }
    public string                ReleaseId   { get; }
    public DeploymentEnvironment Environment { get; }
    public string?               TenantId    { get; init; }
    public bool                  IsCommand   { get; }
    public bool                  IsQuery     { get; }
    public Guid                  CommandId   { get; } = Guid.NewGuid();
}
```

The kill-switch behavior, variant selector, and `IDispatchEvaluationContextProvider` all consume `SharpNinjaFlagCallContext` directly with strong types. No string-keyed `Properties["ProductId"]` lookups. CEL rules see real typed fields.

**Adaptation 2: `MethodInfo.Invoke` replaced with `IHandlerInvoker<TRequest, TResult>`; dispatcher is now AOT-safe.**

`Dispatcher.InvokeHandler` (line 167 of upstream `Dispatcher.cs`) uses `handlerType.GetMethod("HandleAsync")` + `MethodInfo.Invoke` to call into the resolved handler. We replace this with a strongly-typed invoker interface:

```csharp
public interface IHandlerInvoker<TRequest, TResult>
{
    Task<Result<TResult>> InvokeAsync(TRequest request, CallContext context);
}
```

The dispatcher's `DispatchAsync` resolves `IHandlerInvoker<TRequest, TResult>` from DI and calls `InvokeAsync` directly ŌĆö strongly typed, no reflection. The ┬¦6.9 source generators emit one invoker per discovered handler at consumer-build time:

```csharp
// emitted by ┬¦6.9 generator for a handler class CheckoutCommandHandlerV2
internal sealed class CheckoutCommandHandlerV2_Invoker
    : IHandlerInvoker<CheckoutCommand, CheckoutResult>
{
    private readonly CheckoutCommandHandlerV2 _handler;
    public CheckoutCommandHandlerV2_Invoker(CheckoutCommandHandlerV2 handler) => _handler = handler;
    public Task<Result<CheckoutResult>> InvokeAsync(CheckoutCommand request, CallContext context)
        => _handler.HandleAsync(request, context);
}
```

Trim-safe, AOT-safe, faster than reflection. The upstream change to the dispatcher is one method (`InvokeHandler` is replaced with a direct DI resolve plus a typed call). The reflective assembly-scan path in `AddCqrsHandlers` remains for JIT-only hosts that don't want to use generators ŌĆö it now also registers a generic reflective invoker as a fallback. **Hosts that use the generator (which is every host targeting mobile) get AOT-safe dispatch.** The artificial "server-side only" restriction is dropped: the dispatcher runs everywhere the SDK runs.

**Adaptation 3: `IHandlerSelector<TRequest, TResult>` is a first-class dispatcher extension point.**

`AddCqrsHandlers` registers handlers 1:1 via `services.AddTransient(handlerInterface, implType)`. To support variant selection without registering wrapper handlers as the canonical handler, the dispatcher consults an `IHandlerSelector<TRequest, TResult>` before resolving an invoker:

```csharp
public interface IHandlerSelector<TRequest, TResult>
{
    IHandlerInvoker<TRequest, TResult> Select(TRequest request, CallContext context);
}
```

The default registration is `DefaultHandlerSelector<TRequest, TResult>` which returns `serviceProvider.GetRequiredService<IHandlerInvoker<TRequest, TResult>>()`. The flag-aware DI extension registers `FlagAwareHandlerSelector<TRequest, TResult>` for any `(TRequest, TResult)` pair where the ┬¦6.9 generator emitted a `HandlerVariantMap<TRequest, TResult>` singleton:

```csharp
internal sealed class FlagAwareHandlerSelector<TRequest, TResult>
    : IHandlerSelector<TRequest, TResult>
{
    private readonly HandlerVariantMap<TRequest, TResult>   _map;
    private readonly ISharpNinjaFeatureClient                     _flags;
    private readonly IServiceProvider                       _services;

    public FlagAwareHandlerSelector(
        HandlerVariantMap<TRequest, TResult> map,
        ISharpNinjaFeatureClient flags,
        IServiceProvider services)
    { _map = map; _flags = flags; _services = services; }

    public IHandlerInvoker<TRequest, TResult> Select(TRequest request, CallContext context)
    {
        var ctx     = (SharpNinjaFlagCallContext)context;     // typed, per Adaptation 1
        var variant = _flags.GetString(_map.FlagKey, "", ToEvaluationContext(ctx)).Value;
        var chosen  = _map.Resolve(variant);            // returns Type of the concrete invoker
        return (IHandlerInvoker<TRequest, TResult>)_services.GetRequiredService(chosen);
    }
}
```

The dispatcher's resolution path becomes: `_selector.Select(request, context) ŌåÆ invoker.InvokeAsync(request, context)`. Variant selection is first-class. The kill-switch and behavior-flag-gate machinery remain in `IPipelineBehavior` instances, unchanged from upstream's pipeline composition.

**Net upstream surface change:** one keyword removed (`sealed`), one new constructor parameter on `Dispatcher` (`IDispatchContextFactory`), one method on `Dispatcher` rewired (`InvokeHandler` ŌåÆ typed invoker resolve + call), two new public interfaces added to the CQRS package (`IDispatchContextFactory`, `IHandlerInvoker<,>`, `IHandlerSelector<,>` ŌĆö three, counting selector). All adaptations are recorded in `docs/cqrs-adaptation-delta.md` with the exact line-numbered diff against upstream.

#### 6.10.3 Flag-aware extensions layered on top

The flag-aware additions are **additive**, not invasive. The existing dispatcher's core dispatch path is unchanged in v1; flag awareness is introduced as new pipeline behaviors and a new handler-selection callback the dispatcher already supports (or a thin extension point added during adaptation, if the existing dispatcher does not expose a selector hook). The three deterministic steps in v1 are:

1. **Kill-switch behavior.** A `KillSwitchBehavior` registered at the head of the pipeline inspects the dispatched request's type for `[FeatureFlagGate(key)]` and short-circuits per the attribute's `DisabledBehavior` (`Throw` raises a `FeatureDisabledException`; `ReturnDefault` returns `default(TResponse)`; `NotFound` returns a sentinel handled by integration layers; `NoOp` is invalid for non-void responses).

2. **Variant-selection behavior or selector hook.** Multiple handlers registered for the same `IHandler<TRequest, TResponse>` with distinct `[FeatureFlagVariant(key, value)]` attributes are reconciled into a `HandlerVariantMap<TRequest, TResponse>` (emitted by the ┬¦6.9 generators). A `VariantSelectionBehavior` ŌĆö or, preferably, an `IHandlerSelector` extension point on the existing dispatcher ŌĆö uses the map to pick the variant per dispatch. A `[FeatureFlagFallback(typeof(IHandler<TRequest, TResponse>))]` handler is selected if no variant matches.

3. **Pipeline gating.** Behaviors may carry `[FeatureFlag(key)]` at the type level; disabled behaviors are skipped at composition time. This piggybacks on the existing pipeline-composition mechanism rather than introducing a new one.

Adaptation invasiveness ŌĆö additive behaviors versus a deeper change to dispatcher core ŌĆö is decided at the end of ┬¦6.10.2 discovery, not before. If the existing dispatcher already exposes the hooks the additive approach needs, no core change is made. If not, a minimal core change is made and recorded in the adaptation delta.

#### 6.10.4 Evaluation-context enrichment

A new `IDispatchEvaluationContextProvider` is registered into the flag client's evaluation path. On every dispatch, it adds to the `EvaluationContext`:

- `CommandType` ŌĆö the full type name of the dispatched request, e.g. `"Acme.Checkout.CheckoutCommand"`.
- `CommandId` ŌĆö a `Guid` generated per dispatch for correlation.
- `CorrelationId` ŌĆö pulled from the existing dispatcher's request envelope where available, otherwise from `Activity.Current?.RootId`, otherwise equal to `CommandId`. The exact source is determined in ┬¦6.10.2.
- `TenantId` ŌĆö pulled from the existing dispatcher's tenant abstraction if one exists, otherwise from an injectable `ITenantContext` if registered, otherwise null.
- `IsCommand` / `IsQuery` ŌĆö derived from the request marker interface as used by the existing dispatcher.

CEL rules can then target dispatch-level facts directly:

```cel
context.CommandType == "Acme.Checkout.CheckoutCommand"
  && context.AppVersion.semver_satisfies(">=1.4.0")
  && context.IsCommand
```

The enrichment is automatic; consumer code never populates these fields manually. The provider reads from the existing dispatcher's request envelope; the adaptation delta records exactly which fields exist there and which need to be added.

#### 6.10.5 Synergy with ┬¦6.9 source generators

The ┬¦6.9 generators detect handler classes ŌĆö types implementing `ICommandHandler<TCommand, TResult>` or `IQueryHandler<TQuery, TResult>` from the adapted `SharpNinja.FeatureFlags.Cqrs` namespace ŌĆö and emit dispatcher-aware registration extensions in addition to the generic DI extensions in ┬¦6.9.2. For each handler set bearing flag attributes, the generator produces:

```csharp
// developer writes (in their handlers):
[FeatureFlagVariant("checkout.handler.variant", "v1")]
public sealed class CheckoutCommandHandlerV1 : ICommandHandler<CheckoutCommand, CheckoutResult> { ŌĆ” }

[FeatureFlagVariant("checkout.handler.variant", "v2")]
public sealed class CheckoutCommandHandlerV2 : ICommandHandler<CheckoutCommand, CheckoutResult> { ŌĆ” }

[FeatureFlagFallback(typeof(ICommandHandler<CheckoutCommand, CheckoutResult>))]
public sealed class CheckoutCommandHandlerV1 : ICommandHandler<CheckoutCommand, CheckoutResult> { ŌĆ” }   // same as v1

// generator emits:
public static IServiceCollection AddCheckoutHandlersWithFlags(this IServiceCollection services)
{
    // Concrete handlers register as themselves so the selector can resolve them by Type.
    services.AddTransient<CheckoutCommandHandlerV1>();
    services.AddTransient<CheckoutCommandHandlerV2>();

    services.AddSingleton(new HandlerVariantMap<CheckoutCommand, CheckoutResult>(
        flagKey: "checkout.handler.variant",
        fallback: typeof(CheckoutCommandHandlerV1),
        variants: new Dictionary<string, Type>(StringComparer.Ordinal)
        {
            ["v1"] = typeof(CheckoutCommandHandlerV1),
            ["v2"] = typeof(CheckoutCommandHandlerV2),
        }));

    // The interface resolution returns the selector wrapper (see ┬¦6.10.2.3 hot-spot 2).
    services.AddTransient<ICommandHandler<CheckoutCommand, CheckoutResult>,
                          FlagSelectingCommandHandler<CheckoutCommand, CheckoutResult>>();

```

The dispatcher consults `IHandlerSelector<CheckoutCommand, CheckoutResult>` per ┬¦6.10.2.3 Adaptation 3. With the flag-aware DI extension registered, that selector is `FlagAwareHandlerSelector<,>`, which reads the variant map, evaluates the flag against the typed `SharpNinjaFlagCallContext` (per Adaptation 1), and returns the chosen `IHandlerInvoker<,>` (per Adaptation 2). The invoker calls `HandleAsync` directly with no reflection. Trim-safe, AOT-safe, and the dispatcher runs everywhere the SDK runs ŌĆö mobile and server alike.

The same pattern applies for queries: `IQueryHandler<TQuery, TResult>` implementations carrying `[FeatureFlagVariant]` attributes produce `IHandlerInvoker<TQuery, TResult>` invokers, a `HandlerVariantMap<TQuery, TResult>`, and the `FlagAwareHandlerSelector<TQuery, TResult>` registration. The generator emits both command and query variants automatically based on which handler interface the attributed type implements.

#### 6.10.6 Built-in flag-aware behaviors (new additions)

The adaptation adds the following behaviors that did not exist in the upstream system. They implement the existing dispatcher's behavior interface so they compose with everything already there:

- `ExposureLoggingBehavior` ŌĆö emits a structured `Information` log per dispatch with command type, handler type, duration, and outcome. Uses event IDs in the CQRS band (9000ŌĆō9999, per ┬¦7.2 update).
- `FlagSnapshotBehavior` ŌĆö captures the resolved value of an enumerated set of flags at dispatch entry for audit correlation. Gated by `cqrs.flag-snapshot.enabled`.
- `KillSwitchBehavior` ŌĆö implements ┬¦6.10.3 step 1.
- `VariantSelectionBehavior` ŌĆö implements ┬¦6.10.3 step 2 in the additive variant; bypassed if a deeper `IHandlerSelector` hook is wired during adaptation.
- `ConcurrencyLimitBehavior` ŌĆö per-command-type semaphore for known-slow operations. Gated by per-command flag `cqrs.{commandType}.concurrency-limit`.

All behaviors are DI-resident and comply with TR-11 and ┬¦7. The upstream system's existing behaviors are preserved unchanged in v1.

#### 6.10.7 What v1 does not include

No upstream-sync tooling. The vendor copy diverges from McpServer at adoption with the ┬¦6.10.2.3 adaptations applied; reconciliation is a v2 concern (see ┬¦13 open questions).

No event-sourcing primitives. Event sourcing is orthogonal to CQRS-dispatch and would require its own design; the dispatcher remains request/response only in v1.

No saga orchestration. Long-running multi-command workflows are out of scope; they would compose poorly with a flag layer that may flip mid-saga.

No automatic request-batching or fan-out. Single dispatch in, single response out, by design.

No JIT-only fallback mode that bypasses generators. Every host that wants flag-aware CQRS uses the ┬¦6.9 source generators to produce typed `IHandlerInvoker<,>` registrations. Hosts that do not use the generator can still consume the dispatcher through the upstream assembly-scan path with a generic reflective invoker ŌĆö but that path is server-side-JIT only and is documented as such. The recommended path on every platform is generator-emitted invokers.

## 7. Cross-Cutting Concerns: Dependency Injection and Logging

TR-11 makes DI and `ILogger` a uniform contract across every layer. This section is the binding specification for how that contract is realized.

### 7.1 The DI contract

Every public package in the ecosystem ships exactly one `IServiceCollection` extension method that registers that package's services with sensible defaults and an `Action<TOptions>` for tunables. Packages that target MAUI also ship a `MauiAppBuilder` extension that delegates to the `IServiceCollection` variant. The naming convention is `AddSharpNinjaŌĆ”` (`AddSharpNinjaFeatureFlags`, `AddSharpNinjaFeatureFlagsAdmin`, `AddSharpNinjaFeatureFlagsDistribution`, `AddSharpNinjaFeatureFlagsDataPostgres`, etc.).

Internally, every injectable type is registered with the appropriate lifetime ŌĆö singleton for stateful caches and configuration holders, scoped for per-request work in ASP.NET Core and Blazor, transient for stateless helpers. No type is registered with a lifetime that allows captive-dependency hazards (a scoped service injected into a singleton). A `ServiceProviderOptions.ValidateScopes = true` and `ValidateOnBuild = true` check runs in Development environments and in CI to catch these statically.

Constructor injection is the only injection style permitted. Property injection, method injection, and `IServiceProvider.GetService(typeof(T))` resolution from application code are all forbidden. Application code that needs runtime resolution (e.g., factory patterns) injects a typed factory delegate or an `IServiceProvider`-wrapping abstraction registered by the package, never `IServiceProvider` itself.

No public type in any Byrd package has a public constructor that consumers are expected to call. Constructors of injectable types are an implementation detail; the supported way to obtain an instance is `GetRequiredService<T>()` (in test code) or constructor injection (in product code). This is enforced by a public-API test that asserts no public type has a callable parameter-less or new()-friendly constructor outside the schema and options-record types.

### 7.2 The logging contract

Every type that performs logging declares `ILogger<TSelf>` as a constructor parameter and stores it in a `private readonly` field. The category is therefore always the fully-qualified type name ŌĆö no manual category strings. No type uses `ILoggerFactory` directly except the few places that genuinely need to create loggers for dynamically-named categories (none planned for v1).

Structured logging is mandatory. Every log call uses message templates with named placeholders (`logger.LogInformation("Evaluated {FlagKey} for {ProductId}: {Value}", ...)`); string interpolation, `string.Format`, and concatenation into log messages are forbidden because they destroy structured-log fidelity downstream.

Hot paths use the `LoggerMessage` source-generator pattern (`[LoggerMessage(EventId = ŌĆ”, Level = ŌĆ”, Message = "ŌĆ”")] static partial void LogX(...)`) to eliminate per-call allocations and to materialize event IDs at compile time. The rule evaluator, the manifest cache, and the SDK fetcher are the primary hot-path consumers and shall use this pattern.

Log levels follow a strict ladder. `Trace` and `Debug` are for developer diagnostics and shall not be enabled in production by default. `Information` is for operationally significant transitions (manifest refreshed, kill-switch armed, provider initialized). `Warning` is for recoverable anomalies (signature failed, falling back to cache). `Error` is for failures that change observable behavior (cache corrupt, evaluator threw). `Critical` is reserved for unrecoverable subsystem failures.

PII redaction is policy. User IDs and device IDs are hashed with a one-way function (HMAC-SHA256 with a per-deployment salt) before being included in any log at `Information` or higher. Flag keys are not considered PII but rule expressions may contain operator names or comments that should not be exposed; the rule evaluator never logs full rule text above `Debug`. The redaction helpers live in `SharpNinja.FeatureFlags.Diagnostics` and are exposed as a `ILogValueRedactor` service so consumers and tests can substitute.

Event IDs are reserved in 1000-number bands per component (`SharpNinja.FeatureFlags`: 1000ŌĆō1999, `SharpNinja.FeatureFlags.Evaluation`: 2000ŌĆō2999, `SharpNinja.FeatureFlags.Manifest`: 3000ŌĆō3999, Distribution service: 4000ŌĆō4999, admin plane: 5000ŌĆō5999, data layer: 6000ŌĆō6999, CLI: 7000ŌĆō7999, MSBuild integration: 8000ŌĆō8999). Within each band, individual event IDs are stable and documented; renumbering an event ID is a breaking change.

### 7.3 Per-component application of the contract

**Client SDK and provider.** Construction happens through `AddSharpNinjaFeatureFlags`. The provider holds `ILogger<SharpNinjaFeatureFlagProvider>`, the rule evaluator holds `ILogger<CelRuleEvaluator>`, the manifest store holds `ILogger<ManifestStore>`, the fetcher holds `ILogger<DistributionFetcher>` plus a `HttpClient` from `IHttpClientFactory`. The signature verifier holds `ILogger<SignatureVerifier>`. None of these has a public parameterless constructor.

**Rule engine.** The evaluator is registered as a singleton. The parser is registered as a singleton with an internal `ConcurrentDictionary<string, CompiledExpression>` cache. The custom-function table is registered as a singleton; tests substitute it to inject deterministic mocks for `bucket()` and `version_compare()`.

**Manifest store and cache.** Registered as singleton. Reads cache from disk in its hosted-service initialization step; never blocks consumer construction.

**Distribution service.** Standard ASP.NET Core 10 minimal API host. `AddSharpNinjaFeatureFlagsDistribution` registers the manifest origin store, signature signer, exposure-event ingestion handler, and API-key validator. Endpoint handlers accept `ILogger<TEndpoint>` and the relevant services via parameter injection.

**Admin plane (Blazor).** Pages accept `ILogger<TPage>` via `[Inject]`. Application services (`IFlagService`, `IManifestPublishService`, etc.) are scoped per circuit. The dialect helper (`IProviderDialect`) is registered as a singleton resolved from configuration. The migration-history bootstrap that ensures `__SharpNinjaMigrations` exists runs as an `IHostedService` on startup.

**Data layer.** `AddDbContext<AdminDbContext>` registers the context as scoped per request/circuit. Repositories accept `AdminDbContext` and `ILogger<TRepo>` and are themselves scoped. `IDbContextFactory<AdminDbContext>` is registered for the small number of background services that need to outlive a request.

**CLI (`flagctl`).** Built on `Microsoft.Extensions.Hosting`'s generic host so DI and `ILogger` are first-class. Commands are registered services; the command runner resolves the requested command from the container.

**MSBuild integration / source generator.** The source generator runs at compile time and is not a DI participant ŌĆö generators do not run inside the consumer's container. However, the code the generator emits shall be DI-friendly: generated registration extensions, generated `[FeatureFlagProductIdentity]` assembly attribute, and an internal generated `IServiceCollection` extension that registers the embedded bundled-default manifest as a singleton `BundledManifest` value to be consumed by the manifest store. The generator never emits a `static` cache or a global.

### 7.5 AOT and trimming

Both DI and `ILogger` source generators are AOT-friendly in .NET 10. The SDK uses only `GetRequiredService<T>()` and constructor injection ŌĆö never `GetService(Type)` with a runtime `Type` ŌĆö to remain trim-safe. The `LoggerMessage` source generator emits static partial methods that survive trimming and AOT, and is the only logging style permitted in the AOT-targeted SDK assemblies.

## 8. Evaluation Algorithm (Normative)

Given a flag key `K`, a default value `D`, and an Evaluation Context `C` (with `C.ProductId` and `C.ReleaseId` non-null and matching the build stamp):

1. Resolve `M` = current Manifest (in-memory; this never blocks).
2. If `M` does not contain a flag with key `K`, return `EvaluationResult(D, DefaultUsed, null, M.Version)`.
3. Let `F` = `M[K]`.
4. If `C.ProductId` is not in `F.ProductScope`, return `EvaluationResult(D, DefaultUsed, null, M.Version)` and emit a `ProductScopeViolation` diagnostic.
5. If `F.Killed == true`, return `EvaluationResult(F.KilledValue, Killed, null, M.Version)`.
6. For each rule `R` in `F.Rules`, in order:
   1. Evaluate `R.When` against `C`. On evaluation error, log and skip the rule (do not throw to caller).
   2. If `R.When` evaluates to true, return `EvaluationResult(R.Value, RuleMatched, index, M.Version)`.
7. Return `EvaluationResult(F.Default, BundledDefault | RemoteDefault, null, M.Version)` depending on the source of `M`.
8. Emit an exposure event if `F.TrackExposure == true`, buffered and uploaded asynchronously.

This algorithm is the testable specification. The TDD suite begins here.

## 9. Multi-Dimensional Resolution: Product and Release

Product and Release are *not* implemented as special-case columns in flag definitions. They are implemented as **required fields of the Evaluation Context** that rules can reference like any other context field, *plus* they are the **addressing key** of a Manifest.

Concretely:

A Manifest is keyed by `(ProductId, ReleaseId)`. The admin plane composes per-(Product, Release) Manifests from a shared pool of flag definitions. This means a single Flag may exist with different rule sets in different (Product, Release) tuples ŌĆö the admin plane handles this by allowing each Product to override rules per Release while preserving a shared Flag identity.

Within rules, `{ "==": [{ "var": "ProductId" }, "acme-pro" ] }` and `{ "semver_satisfies": [{ "var": "AppVersion" }, ">=1.4.0" ] }` are first-class and used identically. This makes the dimensional model open: adding a third required dimension later is a schema migration, not an architecture change.

## 12. Known Risks and Mitigations

The iOS / AOT / dynamic-evaluation risk is mitigated by mandating a tree-walking interpreter ŌĆö no IL emission, no Roslyn scripting, no runtime compilation. This is enforced by AOT-warning treat-as-error in CI.

The trimming risk for MAUI is mitigated by shipping explicit trimming descriptors and including trim-mode validation in the SDK's own CI.

The key-rotation risk is mitigated by accepting that key rotation requires a new SDK build. Manifests carry a key identifier; SDKs accept any of N embedded public keys, which gives a rolling-window upgrade path.

The flag-namespace-collision risk is mitigated by requiring all flags to declare `ProductScope` explicitly; the admin plane enforces this. There is no implicit global namespace.

The offline-staleness risk is mitigated by per-flag `MaxStaleness` metadata; the SDK surfaces a diagnostic when any flag in the active manifest is staler than its bound, and kill-switchable flags carry tighter bounds.

The rule-engine-as-foot-gun risk (the inherent tradeoff of choosing the full ruleset engine option) is mitigated by (a) choosing CEL, which has formal semantics, no implicit coercions, and proven sandbox safety, (b) forbidding regex and user-defined functions in v1, (c) capping macro iterations per evaluation, (d) the Blazor rule editor's live-preview panel runs every keystroke against synthetic Evaluation Contexts before publish, (e) the mandatory diff-before-publish gate surfaces every flag whose resolved value changes for any saved canary context, and (f) treating publishes as audited deploys subject to the two-person production-publish rule.

The cross-provider JSON-column risk (different SQL dialects expose JSON access through incompatible operators) is mitigated by the data-layer contract in ┬¦6.6.4: application code never writes JSON path filters, anything filterable becomes a typed column, and provider-specific JSON handling is encapsulated in repository implementations and the dialect helper.

The migration-discipline risk (developers bundle multi-table changes into one migration class to save effort) is mitigated by the ┬¦6.6.2 enforcement test that reflects over every migration in every provider assembly and fails the build if any migration touches more than one table. The discipline is also taught by the naming convention: a class named `AddColumn_Flags_ExpirationDate` cannot quietly grow to alter `Releases` without renaming, and renaming triggers review.

The 4NF-decomposition cost risk (more tables, more joins, more verbose repositories) is accepted as the cost of correct normalization. The dominant query patterns are well-known and the join planner on every supported provider handles small-cardinality joins competently. EF Core's `Include` / projection support keeps repository code expressive. The alternative ŌĆö denormalized lists of owners or tags in a parent table ŌĆö would re-introduce update anomalies that the project name itself rejects.

The CQRS-adaptation risk (the vendored copy of `f:\github\McpServer\` brings in hidden assumptions about its host environment or non-DI patterns that violate TR-11 and ┬¦7) is mitigated by the ┬¦6.10.2 structured discovery pass: all violations are catalogued in `docs/cqrs-adaptation-delta.md` and remediated **before** any flag-aware extension lands. Sub-task 6.2 in Phase 6 explicitly gates flag-awareness work on the upstream code passing the same DI/logging conformance checks the rest of the ecosystem honours.

The upstream-divergence risk (the vendored copy diverges from McpServer over time and useful upstream improvements are missed, or McpServer absorbs adaptations that should have been contributed back) is accepted at v1 as the price of unblocking shipping. Reconciliation is an open question in ┬¦13 for v0.2. The discovery delta documents the v1 divergence baseline so future re-sync work has a known starting point.

The handler-variant-explosion risk (CQRS makes it almost too easy to add a fifth variant of `CheckoutCommandHandler` without retiring the prior four) is the flagship feature-creep failure mode this project exists to prevent. Mitigations: ┬¦6.9's `SNFF0008`/`SNFF0010`/`SNFF0011` diagnostics fire on expired, sunsetting, and unused flags at consumer build time, turning stale handler variants into visible build noise; the admin plane's flag-lifecycle UI surfaces handler-variant flags with no recent exposure; the two-person production-publish rule applies to lifecycle transitions, so a flag cannot be silently extended forever without review. Variant-handler explosion is treated as a workflow problem the system actively surfaces, not a technical problem the dispatcher solves.
